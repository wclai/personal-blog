## 一、具備的技術

程式開發不是一名外行人用 AI 一分鐘生成一個看起來很 cool 的網站。真正能被推出市場的系統，除了很好用的功能，還需要考慮效能、資料存放、網路安全和可維護性等多方面，並平衡成本。AI 只能輔助不足，而非可完全依賴的工具。

即使只是練習，最少要有以下認知：

1. **寫程式和 debug 的心態**  
    要閱讀 AI 產出來的程式碼，否則只能對 AI 產出照單全收，在錯誤中輪迴，成品永遠達不到預期效果。
2. **前端、後端和資料庫概念**  
    一般大眾一 click 生成的東西，只有前端，好玩。沒有後端計算和資料存放，絕大部分的需求也無法滿足。
3. **版本控制 (Version control)**  
    即使同一個人寫，也有機會出錯致整套程式垮臺，要用 git 保存安全的版本，和決定正式推出哪一個版本。
4. **什麼是 compile、build、deploy、docker、IP？**  
    不懂的先 Google 或問一問 AI，才計劃是否真的可以推上網。
5. **選擇合適的 AI 工具**  
    如何選擇 AI 工具是另一個大題目，不作討論。只是想提醒免費 AI 工具限制較多，提問太多會卡很久。開始前先評估是否值得付費使用更強的工具。而且請使用「專案模式」 (各 AI 名字不一) 進行開發。

現在靠AI，門檻低了，一般人只要花上一點點時間了解以上項目，就能比前快很多倍建成和運作一個小型網上系統。例如 Paul 此前完全沒有接觸過 Next.js ，但這次也用這語言寫成了。

## 二、整理需求

Project Management 規劃流程包括 Scope Management，收集需求再整理成可行的範圍。

1. **要有基本的想法**  
想做什麼，想解決什麼問題， 出來的畫面設計大概是怎樣？

> Prompt 例子：  
> 「我想建設一個網上會員平台，有哪一些常用的登記和登入流程？」  
> 「我想要簡潔的會員登記頁，有什麼區塊和按鈕？請生成一個模版參考。」

2. **整理有價值的功能**  
中間跟 AI 集思廣益有什麼可行的功能，比較不同的方案，和詢問業界使用的 best practice。經過自己思考是否採納，最後收集成一個功能項目列表。愈複雜的功能，開發的時間也愈長，要決定優先順序，捨棄值價低的念頭。

3. **生成需求文件**  
定好了這一個版本的功能項目，請 AI 生成一份產品需求文件 Product Requirement Document (PRD)。這個基礎極之重要，因為功能之間環環相扣，需要讓 AI 考慮整體需要，預先想好前面的邏輯如何配合後面的功能。例如會員登記要有權限設定，才能規範後續功能誰做到甚麼。如果後來才加入新需求，需要修改早已完成的代碼，會多花很多時間又容易出錯。

## 三、預備技術和工具

1. **VS Code 和 GitHub**  
下載 VS Code (Mac 機可考慮 Xcode)，登記 GitHub 會員，兩者都是開發人員不可或缺的工具。往後要把 VS Code 和 GitHub 連接起來，每完成一個階段的工作則可保存版本。

2. **程式語言及開發軟件**  
按前面所生成的 PRD 及自己的技術程度，詢問 AI 使用甚麼程式語言處理前後端，需不需要和用哪一種資料庫，後續用哪一個網域登記及存放，如何封裝。請 AI 把步驟列出，一步一步安裝套件。

鼓勵過程中多發問，學習和比較不同建議，不要盲目跟從。例如 Paul 會問 AI 為何建議用 React，比較和 Angular 在維護成本上的不同；哪種資料庫的連接及加密法較安全；又因剛好考取了 AWS 證照，正正可實作計算 EC2、RDS 和 S3 不同方案的成本。

3. **資料庫結構**  
最後按 PRD 請 AI 制定好資料庫結構 (DB Schema)，來回修改至定稿。正式開發還會先準備技術文件 Technical Specification，但這部分太複雜，作為練習則省略了。

> Prompt 例子：  
> 「請建議履歷表的資料庫結構。有哪些部分 (例如學歷、工作經驗)？每部分有什麼資料(例如開始及結束日期)？」  
> 「資料庫結構要符合 一個人 每個分部 可以有 零至多則紀錄」  
> 「資料庫表格還要什麼系統欄位才符合標準？」

4. **圖形用戶介面**  
如果初次接觸程式開發，請 AI 建議一些使用圖形用戶介面 (GUI) 的工具，例如用 GitHub Desktop 學習 git，用 DBeaver 處理資料 SQL 查詢等。但後面還是要習慣用純文字打 cmd 執行。

## 四、開發流程

開發習慣、撰寫的次序、AI 擔當的角色等做法，因人而異。以下是 Paul 做法：首先確保 git 和 DB 安裝好，連接上 VS Code 能正常操作，然後循以下流水線，逐個功能項目開發。

_流水線：DB -> API -> UI -> test -> bug fix -> enhance (如需要)_

1. **DB**  
因為處理 DB 的 prisma 套件安裝出現問題，乾脆自己寫 SQL 建立和修改 DB。DB Schema 作為第一個要完成的項目，打後才可以讓 API 進行正確的 DB 讀寫動作。

2. **API**  
雖然同時開發前端 UI 切版和後端 API 開發較有效率，但 AI 寫程式以秒速計算，建議先好好完成 API 的部分。因為 AI 容易有記憶錯誤，記得每次 prompt 都要附上相關 DB Schema，才能確保 API 呼叫正確。

3. **UI**  
如果是簡單的畫面，請 AI 用一致簡潔的設計生成響應式 RWD 前端程式，並連接 API。如果想要特定的設計，可以截取喜歡的網頁為圖片給 AI 參考，但請他配合 PRD 和 API 邏輯進行調整。記憶問題，每次 prompt 最好都附上相關 API 程式碼。

4. **測試**  
如果成功連接 API 及在本地跑起來，就要人手打開網頁測試。如何偵錯、哪裡找到錯誤訊息進行排查，就靠經驗。初學者可以問 AI。

5. **功能迭更**  
不要心急，功能要一個一個地迭更，程式碼檔案要一個一個去生成。一次過生成太多檔案，如果報錯，除錯成本不會比逐個寫的少。搞不好 AI 的想法有分歧，連鎖效應導致整批程式碼都不能用了。

## 五、除錯和修改需求

開發人員每天的工作中，佔最多的不是寫程式，而在寫好的程式中找錯處。 即使有 AI 幫助，Debug 也是這個企劃中時間成本佔比最高的部分。以下是 Debug 經驗分享。

1. **錯誤訊息**  
如果有明顯錯誤訊息，把訊息附在可能出錯的程式碼給 AI 修改，一般一次就能解決。這裡建議要向 AI 強調只能改動最少部分，不能更改任何其他沒有問題的程式碼。

2. **結果不正確**  
如果測試結果不似預期，又沒有錯誤訊息，就要仔細向 AI 說明：  
_- 在哪個位置測試  
- 做了什麼動作 / 輸入內容  
- 預期結果  
- 實際的錯誤結果_

2. **附上程式碼**  
如果不附上出錯的程式碼，就請 AI 直接給出「正確」版本，他絕對非常有可能記錯寫法，把原本正確的部分都弄錯。 Paul 初時誤信 AI 的記憶，因此陷於愈改愈錯的輪迴地獄。

3. **閱讀程式碼**  
鼓勵去閱讀程式碼，嘗試自己找出問題；找 AI 改也要多問原理，請他提供多個解法作比較。這是一個珍貴的實戰經驗。而且 AI 不一定什麼都能解決到，有一次遇到 framework 層級的錯誤，AI 一直跳針給出無效的解答，最後研究代碼後上 stackoverflow 找到解決方法。

4. **修改需求**  
如非必要，不要修改需求。這可能會改到 DB Schema，影響其他已完成沒出錯的功能，而且影響並非即時能察覺到。如果必須要改，請認真理解需求和系統結構，先跟 AI 商討可能受影響的部分，都給 AI 檢查是否會動到，改動最少的部分，事後進行足夠的測試。

**測試計劃 (test plan) 進階使用**  
Project Management 包括 Quality Planning ，應用於開發流程，則可制定標準的測試計劃。可與 AI 合作生成測試步驟，再用人手執行。最簡單的 functional test 例子：  
- 功能項目  
- 參與角色  
- 測試項目位置  
- 進行動作 / 輸入內容  
- 預期結果

## **七、完成後心得整理**

1. 作為 PM 練習，可以把 AI 當成技術高超、見多識廣的 Developer。由於他猜度人心不一定準確，所以 PM 下 prompt 要全面及精準，適時提問及調整。一 click 生成的心態等如叫工程師通靈，出來的東西只是一件能跑的裝飾品。
2. 作為寫程式的練習，讓學習者更快地用實際效果去理解一套語言。由於所作之物不太能成為正式產品，那就不用怕出錯，多問多試多練功。而且不能盡信 AI，要嘗試理解代碼，讓自己能續寫及除錯。
3. AI 另個強大的地方，是在完成開發後，請他按對話紀錄生出各個專案文件，例如重寫 PRD，生成 Data Dialogue、用戶使用手冊及系統宣傳電郵等草稿，大大減省寫文件的時間。
4. 不是說 AI 取代工程師，也不要取笑嘗試 vibe coding 的人無知。AI 確實大大降低了程式開發的門檻，並加快工作速度。即使今天 vibe coding 出來的東西只有雛形，願意嘗試的人會不斷進步，最後由肯創新的取代固步自封的人。
